# Codex `app-server` JSON-RPC Protocol — Phase 1 Findings

Empirical capture for the Codex Phase-1 "pre-connection" refactor (one `codex app-server` subprocess, N threads = N sessions). All data below was produced against the local Codex CLI 0.120.0 on 2026-04-20; the captures are verbatim from live subprocess runs (see `/tmp/codex-probe/probe*.py`).

---

## 1. Invocation

```bash
codex app-server --listen stdio://
```

- Default `--listen` is `stdio://` (confirmed via `--help`); explicit form used for clarity.
- `-c key=value`, `--enable <feature>`, `--disable <feature>` pass through to the TOML config layer. No `-c` flags are required to get turn/start working, but they remain the knob for fine-tuning sandbox / approvals beyond the built-in `approvalPolicy` / `sandbox` params of `thread/start`.
- The server speaks line-delimited JSON-RPC 2.0 on stdin / stdout. Stderr carries `tracing` diagnostics (level controlled by `RUST_LOG`).

The underlying method catalog is generated by macro in `/Users/zal/Cteno/tmp/codex/codex-rs/app-server-protocol/src/protocol/common.rs` (`client_request_definitions!`, `server_request_definitions!`, `server_notification_definitions!`). Payload structs live in `protocol/v1.rs` (legacy) and `protocol/v2.rs` (current / V2 API).

## 2. Handshake — `initialize` + `initialized`

### Request

```json
{"jsonrpc":"2.0","id":1,"method":"initialize",
 "params":{"clientInfo":{"name":"cteno-p1-probe","title":"probe","version":"0.1.0"},
           "capabilities":{"experimentalApi":true}}}
```

- `clientInfo.name` / `title` / `version` are all required.
- `capabilities.experimentalApi: true` is required to unlock `thread/start.*` experimental sub-fields (`dynamicTools`, `experimentalRawEvents`, `persistFullHistory`, etc.). Without it the server rejects those sub-fields.
- `capabilities.optOutNotificationMethods: [string]` lets you silence specific notification wire names (documented but not used here).

### Response (verbatim)

```json
{
  "id": 1,
  "result": {
    "userAgent": "cteno-p1-probe/0.120.0 (Mac OS 26.2.0; arm64) ghostty/1.2.3 (cteno-p1-probe; 0.1.0)",
    "codexHome": "/Users/zal/.codex",
    "platformFamily": "unix",
    "platformOs": "macos"
  }
}
```

(Schema: `v1::InitializeResponse` — `userAgent`, plus undocumented runtime fields `codexHome`, `platformFamily`, `platformOs`.)

### `initialized` notification

```json
{"jsonrpc":"2.0","method":"initialized"}
```

- No `id` — strictly a notification.
- No `params` (the server accepts `params` absent or `null`).
- Server emits **no direct reply**; after this point `thread/start` et al. are callable.

## 3. `thread/start` — opening a new session on an existing connection

### Request

```json
{"jsonrpc":"2.0","id":2,"method":"thread/start",
 "params":{"cwd":"/tmp/codex-probe","approvalPolicy":"never","sandbox":"workspace-write"}}
```

Supported params (subset used; full list in `v2::ThreadStartParams`):

- `model`, `modelProvider`, `cwd`, `approvalPolicy`, `sandbox`, `config`
- `baseInstructions`, `developerInstructions`, `personality`, `ephemeral`
- Experimental (gated by `capabilities.experimentalApi`): `dynamicTools`, `mockExperimentalField`, `experimentalRawEvents`, `persistExtendedHistory`

### Response (verbatim, truncated)

```json
{
  "id": 2,
  "result": {
    "thread": {
      "id": "019da730-e0ba-7503-8caa-41549fe3ee48",
      "forkedFromId": null, "preview": "", "ephemeral": false,
      "modelProvider": "openai",
      "createdAt": 1776626491, "updatedAt": 1776626491,
      "status": {"type": "idle"},
      "path": "/Users/zal/.codex/sessions/2026/04/20/rollout-2026-04-20T03-21-31-019da730-e0ba-7503-8caa-41549fe3ee48.jsonl",
      "cwd": "/tmp/codex-probe", "cliVersion": "0.120.0", "source": "vscode",
      "agentNickname": null, "agentRole": null, "gitInfo": null, "name": null,
      "turns": []
    },
    "model": "gpt-5.4", "modelProvider": "openai", "serviceTier": null,
    "cwd": "/tmp/codex-probe", "approvalPolicy": "never", "approvalsReviewer": "user",
    "sandbox": {"type": "workspaceWrite", "writableRoots": ["/Users/zal/.codex/memories"],
                "readOnlyAccess": {"type": "fullAccess"}, "networkAccess": false,
                "excludeTmpdirEnvVar": false, "excludeSlashTmp": false},
    "reasoningEffort": "high"
  }
}
```

Immediately followed by a `thread/started` notification (duplicate thread payload) and zero-or-more `mcpServer/startupStatus/updated` notifications (connection-scoped; see §7).

**Two `thread/start` calls on the same connection return two distinct `thread.id` UUIDs** — verified in probe2 / probe3. A second session can stream concurrently with the first.

### Is config per-thread or per-connection?

**Per-thread.** `model`, `modelProvider`, `cwd`, `approvalPolicy`, `sandbox`, `config`, `baseInstructions`, `developerInstructions`, `personality` all live inside `ThreadStartParams`. The `initialize` handshake is pure client-info + capability negotiation — it does *not* commit the connection to a model / cwd / policy. This is the enabling fact for one-subprocess-N-sessions.

MCP servers (`config.mcp_servers`) are the exception worth flagging: they are started eagerly by the server at first-thread time (we see `mcpServer/startupStatus/updated` notifications) and are connection-global.

## 4. `thread/resume`

Same shape as `thread/start` but keyed by `threadId`; see `v2::ThreadResumeParams`:

```json
{"jsonrpc":"2.0","id":N,"method":"thread/resume",
 "params":{"threadId":"<uuid>",
           "cwd":"…","approvalPolicy":"…","sandbox":"…"}}
```

Response carries the same `thread` + `model` + `cwd` + `approvalPolicy` + `sandbox` + `reasoningEffort` fields. Resume overrides propagate to subsequent turns.

## 5. `turn/start` / `turn/completed` — streaming a turn

### Request

```json
{"jsonrpc":"2.0","id":10,"method":"turn/start",
 "params":{"threadId":"<uuid>",
           "input":[{"type":"text","text":"Say 'hello from A' and stop."}]}}
```

`turn/start` accepts the same policy-override knobs as `thread/start` plus `effort`, `summary`, `outputSchema`, `collaborationMode` (experimental).

### Immediate response

```json
{"id":10,"result":{"turn":{"id":"019da731-66b4-7a43-b335-06f9fcb6f1b6",
                           "items":[],"status":"inProgress",
                           "error":null,"startedAt":null,"completedAt":null,"durationMs":null}}}
```

The `turn.id` is critical — it is the key you need to call `turn/interrupt` later.

### Streaming notifications (wire-ordered sample)

```
thread/status/changed            { threadId, status.type: "active" }
turn/started                     { threadId, turn }
item/started                     { threadId, turnId, item: {type:"userMessage"} }
item/completed                   { threadId, turnId, item }
item/started                     { threadId, turnId, item: {type:"reasoning"} }
item/completed                   { threadId, turnId, item }
item/started                     { threadId, turnId, item: {type:"agentMessage"} }
item/agentMessage/delta          { threadId, turnId, itemId, delta }   (N×)
item/completed                   { threadId, turnId, item: assistant-final }
thread/tokenUsage/updated        { threadId, turnId, tokenUsage }
thread/status/changed            { threadId, status.type: "idle" }
turn/completed                   { threadId, turn: {status: "completed"|"interrupted"|…} }
```

**Every thread-scoped notification carries `threadId` at the top of `params`.** Most also carry `turnId`. This is the demultiplexing key — all we need for one-connection / N-thread routing.

### Non-thread-scoped notifications observed on a connection

- `mcpServer/startupStatus/updated` — `{ name, status, error }` (no `threadId`)
- `account/rateLimits/updated` — `{ rateLimits: … }` (no `threadId`)

Plus any of: `configWarning`, `deprecationNotice`, `accountUpdated`, `appListUpdated`, `fuzzyFileSearch/*`, `mcpServer/oauthLogin/completed`, various `windowsSandbox/*`. These need a connection-level listener, not a per-thread broker.

## 6. `turn/interrupt` — cancellation is thread-scoped

### Request

```json
{"jsonrpc":"2.0","id":21,"method":"turn/interrupt",
 "params":{"threadId":"<uuid>","turnId":"<turn-uuid>"}}
```

Both fields are required (see `v2::TurnInterruptParams`). **This is the key answer for the Phase 1 decision: interrupt is scoped to `(threadId, turnId)` and does not affect other threads on the same connection.**

### Response

```json
{"id":21,"result":{}}
```

### Follow-on notifications

```json
{"method":"thread/status/changed","params":{"threadId":"…","status":{"type":"idle"}}}
{"method":"turn/completed","params":{"threadId":"…","turn":{"id":"…","items":[],
                                     "status":"interrupted","error":null,
                                     "startedAt":…,"completedAt":…,"durationMs":2006}}}
```

Verified in probe3: interrupting Turn A's long-running count-to-50 yields a `turn/completed` with `status: "interrupted"` on Thread A only. A concurrent turn on Thread B continues uninterrupted.

## 7. Server → Client requests (approval / elicitation)

All live in `common.rs::server_request_definitions!` and their `v2::*Params` types carry `threadId` (+ `turnId`, + `itemId`). Verbatim method names:

| Wire method | Params struct | Response struct |
|---|---|---|
| `item/commandExecution/requestApproval` | `v2::CommandExecutionRequestApprovalParams` | `v2::CommandExecutionRequestApprovalResponse { decision }` |
| `item/fileChange/requestApproval` | `v2::FileChangeRequestApprovalParams` | `v2::FileChangeRequestApprovalResponse { decision }` |
| `item/tool/requestUserInput` *(experimental)* | `v2::ToolRequestUserInputParams` | `v2::ToolRequestUserInputResponse { answers }` |
| `item/tool/call` | `v2::DynamicToolCallParams` | `v2::DynamicToolCallResponse { contentItems, success }` |
| `account/chatgptAuthTokens/refresh` | `v2::ChatgptAuthTokensRefreshParams` | `v2::ChatgptAuthTokensRefreshResponse` |
| `applyPatchApproval` *(v1 legacy)* | `v1::ApplyPatchApprovalParams` | `v1::ApplyPatchApprovalResponse` |
| `execCommandApproval` *(v1 legacy)* | `v1::ExecCommandApprovalParams` | `v1::ExecCommandApprovalResponse` |

Every v2 approval request carries `threadId` in params — the demultiplexer can route these to the correct session's permission broker without ambiguity. `chatgptAuthTokens/refresh` is connection-scoped (no `threadId`); we must handle it at the connection level.

## 8. Subprocess death behavior

- **Closing stdin** (graceful): server drains and exits 0 within milliseconds. Probe4 captured `exit code after stdin close: 0`.
- **Invalid params / bogus `threadId`**: the server returns a JSON-RPC error and **stays alive**. Other threads continue to receive events normally. Probe5: `{"error":{"code":-32600,"message":"invalid thread id: …"}}` then `thread/start` id=100 succeeds on the same process. **`still alive? returncode=None`** → one bad request cannot crash sibling sessions.
- **Model / provider failure on a single turn**: observed as `turn/completed { status:"failed" , error:{…} }` notification scoped to the failing thread. Siblings unaffected (not directly probed but implied by the error-isolation behavior above; worth adding a capture in Phase-B integration tests).

## 9. Multiplexing summary (actionable for adapter design)

| Routing key on inbound frame | Destination |
|---|---|
| Has JSON-RPC `id`, client-initiated response | `pending_requests[id]` → oneshot. Connection-scoped. |
| Has JSON-RPC `id`, server-initiated request (has `method` AND `id`) | Look up `params.threadId` → route to that thread's inbound-request broker. If absent (e.g. `chatgptAuthTokens/refresh`), route to a connection-level broker. |
| Notification (has `method`, no `id`), `params.threadId` present | Route to `threads[threadId].stream_tx`. |
| Notification, no `threadId` | Connection-level listener (for lifecycle / rate-limits / mcp / warnings). |

Methods where `threadId` is absent but payload is still thread-relevant: `thread/started` carries `params.thread.id` (not `params.threadId`) — the adapter must fall back to `params.thread?.id` when `params.threadId` is missing. Happy does the same in `codexAppServerClient.ts`.

## 10. Implication for Phase B

- **Connection ↔ session is safely 1-to-N.** `interrupt` only touches its target thread, errors on one thread do not kill siblings, subprocess exits cleanly on stdin close. The pre-connection refactor can ship with `supports_multi_session_per_process: true`.
- **`open_connection` responsibility**: spawn `codex app-server --listen stdio://`, send `initialize`, await response, send `initialized`. Store `pending_requests` keyed by JSON-RPC id at the *connection* level.
- **`start_session_on`**: send `thread/start`, grab `thread.id` from response, register it in the connection's `threads` map with a fresh `EventStream` channel.
- **`send_message`**: send `turn/start { threadId }`, capture `turn.id`, let the demuxer feed per-thread notifications into that turn's stream until `turn/completed`.
- **`interrupt`**: send `turn/interrupt { threadId, turnId }`. Current turn_id is already tracked on the session today (`SessionRuntimeState.transport_config`-adjacent `current_turn_id` field) — keep using that.
- **`check_connection`**: `try_wait` the child; if still running, consider it healthy. A zero-cost "ping" method does not exist in the 0.120.0 protocol (we scanned the generated method list — no `ping` / `health`); skip an active round-trip and rely on the subprocess handle + last-seen-notification timestamp.

No trait-design friction surfaced; the four new trait methods map cleanly onto the observed protocol.
